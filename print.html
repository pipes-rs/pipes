Design Overview

The Pipes SDK aims to provide a high-quality and complete interface for defining tasks, workflows, and pipelines.

The Pipes SDK is based on the Pipes IDL, an interface definition language (IDL) which is a language-neutral, platform-agnostic framework for defining tasks, workflows, and pipelines. From executing simple shell scripts to orchestrating complex workflows across microservices, Pipes offers a 'batteries-included' approach.

With a single, unified syntax, Pipes streamlines the design, validation, and implementation of scalable and reusable workflows. Coupled with robust tooling and pluggable interfaces, Pipes empowers developers to generate executable code in multiple languages that can execute tasks in any environment, thereby accelerating the automation and deployment of reliable and efficient task-oriented solutions.

Internals

The Pipes SDK is built on parsing expression grammar (PEG) using pest. We're continually iterating on internals as features improve and are introduced.

Code Generation

The Pipes SDK is written in Rust and code is generated from Pipes models, using procedural macros. Diagrams representing pipelines are also generated from models (useful for defining business processes visually).

Pipes Quickstart

Pipes is a language for defining strongly-typed tasks, workflows, and pipelines.

Some simplified example use-cases can be found below:

  • Events
  • CI/CD
  • Business Processes

Events

An example event pipeline:

/// A pipeline to emit notifications for Github issues
pipeline IssueNotification {
    version: "1.0"
    tasks: [
        ListenForGithubIssue,
        Notify
    ]
}

task ListenForGithubIssue {
    command: String
}

task SendEmail {
    command: String
    depends_on: [ListenForGithubIssue]
}

task SlackWebhook {
    command: String
    depends_on: [ListenForGithubIssue]
}

union Notify {
    webhook: SlackWebhook
    email: SendEmail
}

This pipeline defines a workflow to receive notifications when new Github issues have been added.

A pipeline consists of a version (e.g. 1.0), an identifier name (e.g. IssueNotification), and a collection of tasks.

Tasks may be explicitly defined or, as in the case above, may be defined as a union type when one member variant of the tagged union is desired. In this IssueNotification pipeline, tasks that could be carried out are: ListenForGithubIssue and either SendEmailor SlackWebhook. This allows multiple systems to re-use the same model while emitting different events.

CI/CD

Similarly, we can define a pipeline for CI/CD:

/// A pipeline to deploy an application
pipeline DeployApp {
    version: "1.0"
    tasks: [
        Build,
        Test,
        DeploymentTasks
    ]
}

task Build {
    @default("run build")
    command: String
}

task Test {
    @default("run test")
    command: String
    depends_on: [Build]
}

parallel DeploymentTasks {
    tasks: [DeployAPI, DeployFrontend]
    depends_on: [Test]
}

task DeployAPI {
    @default("run deploy")
    command: String
}

task DeployFrontend {
    @default("run deploy --frontend")
    command: String
}

In this CI/CD pipeline, Build builds the code first. Test runs tests after the build. The DeploymentTasks is an aggregate task type parallel which runs both DeployAPI and DeployFrontend tasks in parallel after the tests pass.

The @default trait applies default values for commands; however, these can be overriden in a client's implementation.

Processes

Let's represent a simplified Business Process Model and Notation (BPMN) workflow as a pipeline:

/// A pipeline for order processing
pipeline OrderProcessing {
    version: "2.0"
    tasks: [
        ReceiveOrder,
        ValidateOrder,
        ApproveOrder,
        RejectOrder,
        FulfillmentTasks,
        NotifyCustomer
    ]
    errors: [
        ProcessingError
    ]
}

task ReceiveOrder {
    command: String
}
  
choice ValidateOrder {
    depends_on: [ReceiveOrder]
    condition: IsValidOrder
    then: ApproveOrder
    else: RejectOrder
}

string IsValidOrder

task ApproveOrder {
    command: String
}
  
task RejectOrder {
    command: String
}

parallel FulfillmentTasks {
    tasks: [PackItems, ArrangeShipping]
    depends_on: [ApproveOrder]
}

task PackItems {
    command: String
}

task ArrangeShipping {
    command: String
}
  
task NotifyCustomer {
    command: String
    depends_on: [FulfillmentTasks]
}

catch ProcessingError {
    depends_on: [ReceiveOrder, ValidateOrder, ApproveOrder, RejectOrder, FulfillmentTasks, NotifyCustomer]
    message: String
}

In this BPMN-like workflow:

  • ReceiveOrder receives the order.
  • ValidateOrder checks whether the order is valid or not and then either approves or rejects the order.
  • FulfillmentTasks is an aggregate task which runs packing and shipping tasks in parallel once the order is approved.
  • NotifyCustomer notifies the customer upon successful fulfillment.
  • ErrorHandling is a catch block that handles any errors that occur during the process.

This could generate a BPMN workflow similar to the following:

flowchart LR
    id1((Start)) --> id2(ReceiveOrder)
    id2(ReceiveOrder) --> id3{ValidateOrder}
    id3{ValidateOrder} -- valid --> id4(ApproveOrder)
    id3{ValidateOrder} -- invalid --> id5(RejectOrder)
    id4(ApproveOrder) --> id6(PackItems)
    id4(ApproveOrder) --> id7(ArrangeShipping)
    id6(PackItems) --> id8(NotifyCustomer)
    id7(ArrangeShipping) --> id8(NotifyCustomer)
    id8(NotifyCustomer) --> id10((End))
    id5(RejectOrder) --> id10((End))

Pipes Specification

This is the specification for Pipes, an interface definition language (IDL) and set of tools to build strongly-typed tasks, workflows, and pipelines.

Features

Protocol-agnostic

Pipes is designed to be language and protocol agnostic to describe tasks, workflows, and pipelines capable of running in any environment.

Extensible

Pipes is extended through plugins and meta-model traits. This allows the framework to evolve without breaking changes and provide entirely customizable workflows.

Task-based Reusability

Pipes models are defined by tasks and can be used across services or projects. By defining tasks with actions, dependencies, and additional metadata, Pipes offers a rich and strongly-typed API to execute a sequence of actions (i.e. workflows, pipelines, etc.).
